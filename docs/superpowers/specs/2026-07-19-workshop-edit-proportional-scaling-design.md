# Пропорциональный пересчёт количеств в форме редактирования цеха

**Дата:** 2026-07-19
**Файл:** `resources/views/workshops/edit.blade.php` (только он; create-форма и backend не затрагиваются)

## Задача

При увеличении/уменьшении количества готовой продукции автоматически пересчитывать
сырьё и упаковку пропорционально, чтобы не править каждое число вручную.
UX-паттерн — как в `stone-receptions/edit.blade.php`: `текущее + [дельта] = результат`.

## UI

Новый узел «Итого продукта» в начале конвейера `.pack-flow`, перед блоком «① Сырьё»,
оформлен в стиле остальных `pf-node`. Внутри — по одной строке на каждый продукт,
присутствующий в операции на момент загрузки страницы:

```
ИТОГО ПРОДУКТА
Подоконник 100x30    100 шт  +  [ 20 ]  =  120 шт
```

- Текущее количество — нередактируемый текст (базовое значение).
- Дельта — `input type="number"`, `step="0.001"`, допускает отрицательные значения,
  placeholder `0`.
- Результат — текст; при значении < 0 подсвечивается `text-danger`.
- Нижние блоки «Сырьё / Упаковка / Продукт» остаются полностью редактируемыми —
  для точечных ручных правок.

## Логика пересчёта (только JS, в существующем inline-скрипте)

1. При загрузке страницы у всех строк с количеством (сырьё, упаковка, продукт)
   запоминается базовое значение (`data-base`).
2. Ввод дельты в строке продукта `i`:
   - поле продукта `i` в блоке «③ Продукт» ← `base_i + delta_i`;
   - коэффициент `k = Σ(base + delta) / Σ(base)` по строкам блока «Итого»;
   - каждая строка сырья ← `base × k`, округление до 3 знаков;
   - каждая строка упаковки ← `ceil(base × k)` (штуки, вверх до целого);
   - вызывается существующий `updateTotals()`.
3. Пересчёт всегда идёт **от базы**, не от текущих значений — дельты можно менять
   многократно без накопления ошибок округления.

## Правила и следствия

- Ручная правка сырья/упаковки, сделанная до ввода дельты, **перетирается**
  пересчётом (подтверждено пользователем).
- Строки, добавленные кнопками «Добавить…» после загрузки страницы, в пропорции
  не участвуют (у них нет базы) — количество вводится вручную; пересчёт их не трогает.
- Строка с базой 0 не масштабируется (умножение на k даёт 0) — корректно само по себе.
- Если сумма базовых продуктов равна 0, пересчёт сырья/упаковки не выполняется.
- `manual_processing_sum` (₽ за единицу) при масштабировании не меняется.
- Удаление строки продукта из нижнего блока скрывает/деактивирует её строку
  в «Итого» и исключает из суммы коэффициента.

## Отправка формы

Без изменений: на сервер уходят те же поля `products[i][quantity]`,
`raw_materials[i][quantity]`, `packages[i][quantity]` — backend, валидация
и `WorkshopService` не модифицируются.

## Тестирование

Чисто фронтенд-логика в Blade-шаблоне — Pest-тесты не требуются.
Проверка вручную: ввод положительной/отрицательной дельты, несколько продуктов,
округление упаковки вверх, ручная правка после пересчёта.
